# Tango Bug

End-to-end bug fix with outside review at the plan stage and again at the
diff stage. Each review is a loop: fix the P0/P1 findings Codex flags,
re-review, repeat until none come back, then move on.

The whole point is to catch regressions, AI slop, and over-engineering that
Claude alone tends to miss — so the review framing pushes hard against
those, not just at correctness.

## Severity & confidence convention (used in §3 and §6)

Codex output follows the official Codex review schema: priority is an
integer 0-3 (P0 most severe), confidence is a float 0.0-1.0, code locations
include file path and line range. We ask for findings in this format:

```
[P0|P1|P2|P3] (confidence: 0.NN) {file}:{start}-{end} — {title}
{body — one paragraph explaining the issue and what to change}
```

Priority meanings:
- **P0** — catastrophic. Security, data loss, fundamentally wrong root cause,
  the change clearly makes things worse. Must fix.
- **P1** — serious bug or regression risk. Functional incorrectness, broken
  callers, missed verification. Must fix.
- **P2** — should fix soon. Code smells, minor edge cases in non-critical
  paths, AI slop that's worth pruning but doesn't break anything.
- **P3** — nit. Style, naming, optional polish.

**Loop gate:** any P0 or P1 finding → fix, re-review. Only P2/P3 (or none) →
exit the loop. P2/P3 findings get logged to the plan's Open Questions and
left for the user to triage.

For code review (§6 only), Codex also returns an `overall_correctness`
verdict: `"patch is correct"` or `"patch is incorrect"`, with its own
justification and confidence. `"patch is incorrect"` always loops back,
regardless of priority labels.

## 0. Setup

Detect the project's conventions up front so later stages can use them:
read `CLAUDE.md`, `README.md`, and any `AGENTS.md` if they exist; look for
formatter/linter config (`.prettierrc`, `pyproject.toml`, `eslint.config.*`,
`Makefile`, `package.json` scripts, etc.). Note what you find — you'll
reference it in §5 (implement) and §7 (sanity check).

If you can't tell what the formatter/linter is, ask the user once. Don't
guess and don't run random commands.

## 1. Investigate → root cause

Investigate normally: read the code, trace the call site, check `git log` in
the implicated area, reproduce if you can. The output of this stage is a
specific code-level claim — `file.ext:line`, a function name, the condition
that produces the symptom — backed by evidence, not speculation.

Bug reports are usually one or two lines. Don't expect a reproducer; produce
your own. Look for an existing test you can extend, or the smallest script
that triggers the symptom — whichever is cheaper.

Also look at the **existing test coverage** for the affected code path: is
there already a test that *should have* caught this? If yes, why didn't it
— is the case missing, the assertion too loose, the test not run? If there
is no test for this code path at all, note that — §2's Verification section
will plan whether to add one.

Ask the user when something critical is missing (no symptom, no hint where
the code lives, ambiguous behavior). One question at a time, plain text, no
multiple-choice unless the choice is genuinely close-ended.

## 2. Write the plan

Write to `./bug-{branch}-{YYYYMMDD-HHMMSS}.md` at the repo root.

```bash
DATETIME=$(date +%Y%m%d-%H%M%S)
BRANCH=$(git branch --show-current 2>/dev/null || echo main)
```

Template — omit any section that's genuinely N/A:

```markdown
# Bug: {one-line title}

Generated by /tango-bug on {date}
Branch: {branch}
Status: DRAFT

## Symptom
What's broken, expected vs actual, error message verbatim if there is one.

## Reproduction
Step-by-step recipe, frequency (always / sometimes / once), environment.

## Root cause
The specific code path that produces the symptom (file:line, function),
with the evidence that points there.

## Recommended fix
What to change and why. If this is a workaround, label it as such.
Keep the fix minimal — don't bundle refactors, don't add error handling
for cases that can't happen, don't introduce abstractions for hypothetical
future requirements.

## Alternatives considered
At least one other approach you ruled out, with a one-line reason.

## Verification
Exact repro steps that should now pass. Adjacent functionality worth
regression-checking.

**Regression test plan.** Decide one of:
- **Add new test** — if no existing test covers this bug AND a unit test
  can cleanly capture it. State the file path and the specific
  assertion(s) that would fail on the unfixed code and pass after the
  fix.
- **Extend existing test** — if a nearby test exists but missed this
  case. State which test and what to add.
- **No new test** — if the bug isn't unit-testable (real I/O, timing,
  flaky external service, UI-only behavior). State the reason
  explicitly so reviewers can challenge it.

## Risk
What could break, blast radius, rollback story.

## Open questions
Anything unresolved. Codex DEFERRED comments land here too.

## Codex plan review
_(filled in by §3)_

## Codex code review
_(filled in by §6)_
```

Tell the user the path and proceed to §3.

## 3. Codex reviews the plan — LOOP

This is a loop, not a single pass. Iterate until Codex returns no P0/P1
findings, or you hit the iteration cap.

### One iteration

Dispatch the `codex:codex-rescue` subagent with the **current** plan as
input. Codex hasn't seen the conversation, so its read is independent.

Call the Agent tool with `subagent_type: "codex:codex-rescue"` and this
prompt:

```
--effort high --fresh

You are reviewing a bug-fix plan written by another engineer. You have NOT
seen their investigation — only the plan below.

Review with these priorities, in order:
1. REGRESSION RISK — will this fix break something that currently works?
   Adjacent code paths, callers of the changed function, shared state.
2. ROOT CAUSE — is the diagnosed cause actually the cause, or just a
   correlated symptom? Name a specific alternative if you see one.
3. SIMPLICITY — could this fix be smaller? Is it doing more than the bug
   requires?
4. AI SLOP — flag anything that smells like AI over-engineering:
   defensive null-checks for cases that can't happen, error handling that
   hides the bug, comments explaining what code already says, refactors
   bundled with the fix, premature abstractions, new helpers used once,
   feature flags for backwards-compat that isn't needed.
5. VERIFICATION — does the verification plan actually prove the fix works?
   Will it catch a regression?

For each finding, emit a single block in this exact format:

[P0|P1|P2|P3] (confidence: 0.NN) {file}:{start}-{end} — {title ≤ 80 chars}
{body: one paragraph explaining the issue and what to change. No preamble.
No location restated. No closing summary.}

Priority meaning:
- P0 = catastrophic (security, data loss, fundamentally wrong root cause)
- P1 = serious (real bug, regression risk, missed verification)
- P2 = should fix (code smell, minor edge case, AI slop worth pruning)
- P3 = nit (style, naming, optional polish)

Confidence is 0.0-1.0. If you're speculating, say so with a low score —
don't inflate confidence to seem certain.

For findings about the plan structure rather than code, use the plan
filename as the file and the line range of the relevant section.

If you find nothing at P0 or P1, return exactly: NO P0/P1 FINDINGS
(followed by any P2/P3 findings if you have them).

PLAN
====
{paste the full plan doc verbatim}
```

Paste Codex's output verbatim under `## Codex plan review` in the plan,
appending it as a new dated subsection (`### Iteration 1 — {timestamp}`,
`### Iteration 2 — ...`) so the audit trail builds up across iterations.

### Decide

- **No P0 or P1 findings** → exit the loop. Append any P2/P3 findings to
  Open Questions with a `[Codex P2]` / `[Codex P3]` tag. Proceed to §4 then §5.
- **One or more P0/P1 findings** → fix them. For each, edit the plan to
  address it. Then run another iteration of §3.

### Loop guardrails

- **Cap at 4 iterations.** If P0/P1 findings still come back after 4 rounds,
  stop and surface the disagreement to the user: paste the remaining
  findings and ask whether to override (proceed anyway), revise further, or
  abandon.
- **Watch for thrashing.** If the same P0/P1 finding comes back two
  iterations in a row, your fix isn't landing — stop iterating and ask the
  user.
- **Ask before scope expansion.** If a finding would meaningfully expand
  the fix (touches new files, changes the chosen approach), confirm with
  the user before applying it.
- **Low-confidence P1.** If a P1 finding has confidence < 0.5, treat it as
  a verification task rather than an automatic fix: investigate first, then
  either (a) confirm and fix, or (b) push back in the plan with evidence
  for why Codex was wrong, and let the next iteration confirm.

## 4. Lock the plan

Once §3 exits clean:

- Update `Status:` from `DRAFT` to `APPROVED`.
- Under `## Codex plan review`, add a one-line summary at the top:
  *"Approved after N iteration(s). Final review: NO BLOCKERS."*

## 5. Implement

Apply the recommended fix as written in the plan, **plus the regression
test the plan specified** (if any). Land the test and the fix in the same
diff so Codex reviews them together — and so the test demonstrates the
fix actually works.

Suggested order:
1. Write the new test (or extend the existing one) first. Run it; confirm
   it fails on the unfixed code with the symptom you expect.
2. Apply the fix. Run the test; confirm it now passes.
3. Run the rest of the project's test suite to confirm no regression.

If the plan said "no new test" with a justification, skip step 1 — but
double-check the justification still holds after seeing the fix; sometimes
the fix opens up a unit-testable surface that wasn't there before.

Keep the diff minimal:

- Don't bundle refactors, formatting changes, or unrelated fixes.
- Don't add error handling, null-checks, or fallbacks for cases that can't
  happen — only validate at real boundaries (user input, external APIs).
- Don't add comments explaining what well-named code already says. Only
  add a comment when the *why* is non-obvious.
- Match the project's existing patterns and conventions (per the §0 read).
- Don't introduce new dependencies unless the plan called for it.

If implementation reveals the plan was wrong (the root cause is somewhere
else, the fix doesn't work), stop. Update the plan with what you learned
and loop back to §3 — don't ship a fix that diverged from a reviewed plan
without re-review.

## 6. Codex reviews the diff — LOOP

Same loop shape as §3, but reviewing the actual code change. Plus an
`overall_correctness` verdict on the whole patch.

### One iteration

Capture the diff:

```bash
git diff           # unstaged
git diff --cached  # staged
# or both: git diff HEAD
```

Dispatch `codex:codex-rescue` with this prompt:

```
--effort high --fresh

You are reviewing a code change that implements a bug fix. Review with these
priorities, in order:
1. CORRECTNESS — does this actually fix the bug as described in the plan?
2. REGRESSION RISK — what existing behavior could this break? Callers,
   shared state, adjacent code paths.
3. SIMPLICITY — is the diff bigger than it needs to be? Anything to cut?
4. AI SLOP — flag defensive code for impossible cases, comments that
   restate code, unused helpers, over-eager error handling, abstractions
   added "just in case".
5. PROJECT CONVENTIONS — does this match the surrounding code's style and
   patterns?

For each finding, emit a single block in this exact format:

[P0|P1|P2|P3] (confidence: 0.NN) {file}:{start}-{end} — {title ≤ 80 chars}
{body: one paragraph. No preamble, no location restated, no closing summary.}

Priority meaning:
- P0 = catastrophic (security, data loss, breaks production)
- P1 = serious (real bug, regression risk, fix doesn't actually fix the bug)
- P2 = should fix (code smell, minor edge case, AI slop worth pruning)
- P3 = nit (style, naming, optional polish)

Confidence is 0.0-1.0. Be honest — speculation should score low.

After all findings, emit the overall verdict on its own line:

OVERALL: patch is correct | patch is incorrect (confidence: 0.NN)
{one-paragraph justification — why it does or doesn't actually implement
the plan and fix the bug}

If you have no P0/P1 findings AND the patch is correct, your output may be
just the OVERALL line. P2/P3 findings are still welcome — list them above
the OVERALL line.

PLAN (for context — this is what the diff is meant to implement)
================================================================
{paste the approved plan doc verbatim}

DIFF
====
{paste the full git diff verbatim}
```

Paste output under `## Codex code review` in the plan, again as a new
`### Iteration N — {timestamp}` subsection.

### Decide

Loop again if **any** of these are true:
- One or more P0 or P1 findings.
- `OVERALL: patch is incorrect`.

Otherwise, exit. Append P2/P3 findings to Open Questions with a `[Codex P2]`
/ `[Codex P3]` tag. Proceed to §7.

### Loop guardrails

Same as §3: cap at 4 iterations, watch for thrashing, ask before scope
expansion. Additionally:

- **`patch is incorrect` overrides priority counts.** Even if the priority
  blocks all came back as P2 (which is unusual when correctness fails), the
  `OVERALL` verdict is the gate — if Codex says the patch doesn't fix the
  bug, loop and re-investigate. The plan may be wrong, not just the code.

## 7. Sanity check

Run the project's formatter and linter as identified in §0. Examples
(use whatever the project actually has, not these literally):

```bash
# Python
ruff check . && ruff format --check .
# JS/TS
pnpm lint && pnpm format:check
# Go
go vet ./... && gofmt -l .
```

Fix anything that fails. If formatter/linter changes are non-trivial (more
than mechanical reformatting), loop back to §6 — Codex hasn't reviewed
those changes yet.

Verify the fix per the plan's Verification section: run the repro, confirm
the symptom is gone, run the new regression test (if the plan added one),
and run the rest of the project's test suite to confirm no adjacent
behavior regressed.

## 8. Fallbacks

If `codex:codex-rescue` is unavailable or errors in §3 or §6, fall back to
dispatching a Claude subagent (Agent tool, no `subagent_type`) with the
same prompt. Rename the relevant section to `## Subagent plan review` /
`## Subagent code review` so the audit trail is honest. Independence still
holds; the only loss is the cross-model perspective.

If that also fails, write one line in the section explaining what happened
and surface to the user — don't silently skip review.

## 9. Done

Update the plan's `Status:` to `IMPLEMENTED`. Tell the user:

- Final plan path.
- Plan review: N iteration(s). Final P2/P3 count if any (P0/P1 is always 0
  by the time we exit).
- Code review: M iteration(s). Final `OVERALL: patch is correct (confidence: …)`
  and any P2/P3 logged to Open Questions.
- Sanity check: pass / what was fixed.
- One-line summary of what shipped.

Don't commit, push, or open a PR unless the user asks.
